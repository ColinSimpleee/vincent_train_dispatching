# 前后端拆分与治本设计文档

**日期**：2026-04-26
**项目**：Headway（列车调度游戏 v2.0）
**作者**：Colin Lee + Claude（brainstorming）
**状态**：待实现计划

---

## 1. 背景与目标

### 1.1 现状

- 纯前端 Vue 3 + TypeScript 项目，无后端、无数据库、无账号体系。
- 物理模拟（60 ticks/s）、渲染、输入、调度业务全部混在 `App.vue`（1206 行）和 `PhysicsEngine.ts`（411 行）中。
- 车站拓扑、时刻表、列车数据全部硬编码在 `src/data/stations.ts`（262 行）。
- 测试覆盖很薄（仅 `ScheduleManager` 与 `useDispatchLog`）。

### 1.2 核心痛点

玩家偶发性体验问题，**且控制台无任何报错**：

1. 列车闯红灯
2. 列车原地不动
3. 列车出站后不消失

这些问题反复修过仍偶发出现。

### 1.3 真实病因诊断

经过讨论，确认这些 bug 不是"前端 vs 后端"问题，而是：

- 状态机缺少不变量断言（错误状态组合静默继续）
- 不可复现（`Math.random()` 无 seed）
- 控制流分散（App.vue 950 行混合定时器/生成/发车/移除）
- 没有结构化日志（出 bug 时控制台空白）

把物理引擎用 Python/C 重写并不能解决这些问题，只会换地方出现。

### 1.4 目标

本次改造覆盖三类需求：

1. **治本**（最高优先级）—— 解决偶发 bug
2. **持久化** —— 玩家进度、历史、排行榜
3. **扩展位** —— 为未来 6 类扩展（不含多人对战）留好接口

明确**不做**：

- 多人实时对战（用户已确认不做）
- 后端跑权威实时物理（违背"前端零延迟操作"前提）
- 第三方托管认证（Auth0/Supabase 等，国内访问问题）

### 2.2 职责划分

| 职责 | 归属 |
|---|---|
| 物理模拟、碰撞、信号、道岔 | 前端 |
| 渲染、动画、音效、玩家输入 | 前端 |

### 2.3 路线时间线

- **阶段一（约 2–3 周，纯前端治本）**：seed / 不变量 / replay / App.vue 拆解
- **阶段二（约 2–3 周，加后端）**：登录 + 存档 + 排行 + 举报 + 配置下发
- **后续扩展**：扩展项 1/2/3/6/7/8 按需迭代（接口位本次留好）

### 2.4 核心设计原则

- **前端管"权威实时状态"，后端管"权威持久状态"**，互不抢
- **阶段一不依赖阶段二** —— 阶段一完成后前端可独立运行
- **数据契约先行** —— 局记录格式（seed + 输入流 schema）阶段一就定下
- **一份引擎源代码** —— 前端和 sidecar 共用 `packages/engine/`，永不飘移

## 3. 阶段一：前端治本

### 3.1 物理引擎纯函数化

把 `PhysicsEngine` 抽成 `packages/engine/` 零依赖纯模块：

```ts
export interface EngineState {
  tick: number
  trains: TrainPhysics[]      // 不再带 Vue reactive 标记
  railMap: RailMap
  rng: PRNGState              // 显式 PRNG 状态
}

export interface EngineInput {
  tick: number
  type: 'admit' | 'depart' | 'stop' | 'switch_toggle' | 'signal_toggle'
  payload: Record<string, unknown>
}

export function step(
  state: EngineState,
  inputs: EngineInput[],
): { next: EngineState; events: EngineEvent[]; violations: Invariant[] }
```

**约束**：
- 不引用 Vue
- 不引用浏览器 API
- 不引用 `Math.random` / `Date.now`
- 相同 `(state, inputs)` 必出相同 `next`

### 3.2 PRNG + Seed

- 算法：mulberry32 或 xoroshiro128+
- 所有随机源都从 `state.rng` 取（列车生成时刻、停站时长、起始游戏时间）
- 每局开局生成 32 位 seed，写进局记录
- bug 出现时 `seed + 输入流` 100% 可复现

### 3.3 不变量断言

每个 tick 末尾跑 `assertInvariants(state)`，开发模式 `throw`，生产模式记录到事件流。

最少覆盖：

| 现象 | 不变量 |
|---|---|
| 闯红灯 | 列车通过节点 N 时 N.signal=red |
| 原地不动 | state=moving 但连续 N tick speed=0；或 stopped 但 boardingTimer=0 且 passengerState=READY 且 signal=green 持续 N tick |
| 出站不消失 | isHandedOver=true 持续 > 60s 仍未从 state.trains 移除 |
| 通用 | path 长度为 0 但 state≠stopped；occupiedBy 与 train.currentEdge 不一致；direction ∉ {-1,1} |

至少 8 条不变量。

### 3.4 结构化事件流

替代 `console.log`：

```ts
type EngineEvent =
  | { tick: T; kind: 'spawn'; trainId; modelType; entrance }
  | { tick: T; kind: 'admit'; trainId; edgeId }
  | { tick: T; kind: 'enter_edge'; trainId; from; to }
  | { tick: T; kind: 'signal_pass'; trainId; nodeId; signalState }
  | { tick: T; kind: 'dwell_start'; trainId; edgeId; duration }
  | { tick: T; kind: 'dwell_end'; trainId; edgeId }
  | { tick: T; kind: 'depart'; trainId; strategy }
  | { tick: T; kind: 'handover'; trainId; edgeId }
  | { tick: T; kind: 'remove'; trainId; edgeId }
  | { tick: T; kind: 'invariant_violation'; rule; detail }
```

环形缓冲区保留最近 N 万条；一键导出 JSON。

### 3.5 Replay 工具（前端 dev-only 面板）

- 拖入 `seed + 输入流 + 期望事件流` JSON
- 按播放：重新初始化引擎 → 按 tick 喂输入 → 实时对比事件流
- 第一处分叉点高亮停下
- **这是定位偶发 bug 的核武器**

### 3.6 App.vue 拆解

```
src/
├── packages/engine/              # 3.1 纯模块
├── game/
│   ├── GameLoop.ts               # tick 推进、倍速、暂停
│   ├── TrainSpawner.ts           # 队列管理
│   ├── DispatchController.ts     # ADMIT/DEPART/STOP 业务逻辑
│   ├── InputRecorder.ts          # 采集输入流
│   ├── EventBus.ts               # 引擎事件 → UI 订阅
│   └── ReplayPlayer.ts           # 3.5 工具
├── components/                    # 渲染层只读 EngineState
│   ├── GameCanvas.vue
│   └── panels/
└── App.vue                        # 组合根，目标 < 200 行
```

UI 层用 Vue reactive 包一层 EngineState 副本；引擎本身不感知 Vue。

每个模块目标 < 300 行。

### 3.7 测试策略

- 引擎模块强制 TDD
- 单 tick 黄金路径测试
- 闯红灯回归测试（fixed seed 复现历史偶发场景）
- 不变量自我测试（喂违法状态，断言能抓到）
- **模糊测试**：100 个随机 seed × 每局 10 万 tick，断言永不违反不变量
- 覆盖率目标 ≥ 80%

### 3.8 阶段一交付物

- [ ] `packages/engine/` 纯模块 + 单测覆盖率 ≥ 80%
- [ ] PRNG 全面替换 `Math.random()`
- [ ] 不变量断言（≥ 8 条）+ 开发模式 throw
- [ ] 事件流环形缓冲 + 导出按钮
- [ ] Replay 面板（dev-only）
- [ ] App.vue 拆为 6 模块，每个 < 300 行
- [ ] 模糊测试 CI 跑 100 局通过
- [ ] **当前已知偶发 bug 至少 1 例被这套工具定位 + 修复**（验收金标准）

### 4.4 认证方案

- 密码哈希：argon2id（passlib）
- Token：JWT，access 15 分钟 + refresh 14 天，refresh 入 DB 可撤销
- 邮箱验证 / 重置：单次签名 token（itsdangerous），有效期 1 小时
- 第三方登录（扩展 8）：第一版仅留位（`password_hash` 已 nullable + `external_identities` 表已建）
- 限流：登录 / 注册 / 密码重置端点 slowapi 按 IP 5 次/分钟

### 4.5 上报流程

```
前端：
  局结束 → InputRecorder 输出 { seed, engineVersion, stationId,
                                 stationVersion, inputs[], summary }
        → HMAC(client_secret, ...) 算 signature
        → POST /api/v1/records  { summary, signature } + multipart inputs.gz

后端：
  ① 验签（client_secret 登录时下发，存内存）
  ② 基础校验（duration / score 在物理上下限、操作频率 ≤ 人类极限、
              stationVersion 在白名单、client_version 在白名单）
  ③ 通过 → 200 { recordId }
  ④ 不通过 → 422 { reason }
```

`client_secret` 存内存而非 localStorage，能挡住 devtools 篡改类攻击；挡不住逆向客户端，依赖 § 5 的举报机制兜底。

### 4.6 配置版本对账

- 前端启动拉 `/configs/stations`，缓存到 IndexedDB，按 `(station_id, version)` 当 key
- 上报时把 `stationVersion` + `engineVersion` 一起报
- 举报回放必须用同版本配置 + 同版本引擎跑

## 5. 排行榜 + 举报 + 调查工作流

### 5.1 排行榜

**计分**：每个车站独立排行榜，`score` 在引擎内计算（举报回放可完整复现）。具体公式留给玩法迭代。

**端点**：

```
GET /api/v1/leaderboard/{station_id}
  ?period=all|monthly|weekly
  ?page=1&size=50
  ?include_reported=true|false   # 默认 true
  ?only_verified=true|false      # 默认 false
```

**返回每行字段**：
```
{ rank, username, score, duration, ended_at, record_id,
  report_status: 'none' | 'pending' | 'investigating' }
```

**展示规则**：

- 默认全部上榜（不因举报暂时下架）
- 每条记录附带 `report_status` 标识：
  - `none` → 无标识
  - `pending` → 🚩 待核查（橙色徽标）
  - `investigating` → 🔍 调查中（黄色徽标）
  - `confirmed_cheat` → 不显示（结案时 `game_records.status='hidden'`）
- 用户可勾选筛选器：
  - "显示待核查记录"（默认开）
  - "显示调查中记录"（默认开）
  - "仅显示已核实记录"（默认关）

**上榜门槛**：
- `result='completed'`
- `duration_ticks >= MIN_PLAYABLE_DURATION`
- 一个用户在一个车站只取本人最高分（`DISTINCT ON (user_id)`）
- `status='visible'`

### 5.2 玩家举报

**入口**：排行榜每行 "🚩 举报" 按钮 → 弹窗：

```
你认为这条记录有问题吗？
□ 完成时间快得不合理
□ 看回放时操作不像人类
□ 其他（必填理由，最多 200 字）
[ 取消 ] [ 提交举报 ]
```

**约束**：
- 必须登录
- 一个 `(record_id, reporter_id)` 只能提交一次
- 速率限制：每人 24h 最多 10 条

### 5.3 管理员调查面板

**布局**：

```
Record #12345    举报 #88
玩家 / 车站 / 引擎 / 得分 / 时长 / 举报理由

自动检查（页面打开自动跑）：
  ✅/❌ HMAC 签名有效
  ✅/❌ 时长 < 物理下限
  ✅/❌ 操作频率合理
  ⚠️  分数高于历史 P99.9
  ⚠️  同 IP 24h 内多账号上分
  ⚠️  client_version 不在白名单

[ 在浏览器中回放 ]    [ 服务端校验回放 ]

服务端校验结果（点击后跑完才出）：
  服务端复算分数 vs 客户端上报分数
  字段不匹配即作弊证据

结案：
  ○ 维持记录（举报无效，status='dismissed'）
  ○ 确认作弊：仅删该条记录
  ○ 确认作弊：删记录 + 用户警告
  ○ 确认作弊：删记录 + 封号 N 天
  备注：[                          ]
  [ 提交结案 ]
```

**自动检查 6 条规则**（具体阈值留给实现期定）：

| 规则 | 数据来源 |
|---|---|
| HMAC 签名无效 | `client_signature` |
| 时长 < 物理下限 | 各车站常量（离线用 sidecar 跑出极限） |
| 操作频率超人类极限 | inputs 流统计 |
| 分数 > 历史 P99.9 | `game_records.score` 分位数 |
| 同 IP 24h 内多账号上分 | 登录 IP 日志 |
| `client_version` 不在白名单 | 上报字段 |

### 5.4 服务端校验回放

按下"服务端校验回放"按钮：

1. 拉 `record.seed` + `inputs.gz` + `stationVersion` + `engineVersion`
2. spawn 子进程：`node sidecar/replay.ts --record=12345`
3. sidecar 用对应版本 `packages/engine` + 对应版本配置，从 seed 初始化，按 tick 喂输入
4. 输出 `{ score, duration_ticks, trains_handled, collisions, invariant_violations }`
5. 与客户端上报字段对比，任意字段不匹配即作弊证据

sidecar 不常驻，按需 spawn 跑完退出。

### 5.5 结案处置

| 选项 | 数据库动作 |
|---|---|
| 维持记录（举报无效） | `reports.status='dismissed'` |
| 确认作弊：仅删记录 | `reports.status='confirmed'` + `game_records.status='hidden'` |
| 确认作弊：删记录 + 警告 | 上 + 邮件 / 站内信通知用户 |
| 确认作弊：删记录 + 封号 | 上 + `users.status='banned'` |

`game_records.status='hidden'` 后，所有排行榜视图自动过滤。

### 5.6 隐私

- 排行榜默认显示 username
- 用户可勾匿名（显示 `Player#####`）
- 输入流默认私有；上榜后管理员可读，举报通过后举报人可读
- 举报理由文字仅管理员可见

### 5.7 自动定期抽查（二期，仅留位）

每周 cron：随机抽 1% 排行榜前 100 跑 sidecar 校验，结果不一致自动建 `reports.status='investigating'` 记录。第一版不实现，但表结构允许。

## 9. 风险与缓解

| 风险 | 缓解 |
|---|---|
| 阶段一抽引擎时引入新 bug | 模糊测试 100 局 + 历史回归测试套件 |
| 阶段二上线后老玩家数据无处迁 | 阶段二上线前没有持久化数据，无迁移问题 |
| 数据库 blob 表撑爆 | 第一版每局 inputs gzip 后 1–10KB，预计千条用户数据约百 MB；超阈值再迁 MinIO |
| HMAC client_secret 泄露 | 接受这层防御只挡 95% 脚本党；强对抗依赖举报+回放 |
| 引擎版本与配置版本组合爆炸 | sidecar 启动时按版本号加载对应包；若旧版本太多可定期归档老 record |
| 阶段二工期超预期 | 必做范围已最小化；2/3/7/8 都是第二迭代 |
